The mental model that wins "best answer" questions

  1. Match the stated priority. The executive statement ranks the real goal — cost, reliability, compliance, speed. Serve it.
  2. Prefer the most managed option that meets the requirement ("minimize operational overhead" → serverless/managed).
  3. Don't over-engineer. Single-region need → don't pick the global/expensive service (Spanner is the classic cost trap).
  4. Eliminate two obviously wrong, then decide the last two on priority.

Domain · Compute

Compute

Pick the runtime by trading control against operational overhead. Choose the least-ops option that still meets the requirement.

Decision tree

Containerized?
├─ NO
│  ├─ OS control / licensing / lift-and-shift / custom kernel ─→ Compute Engine
│  ├─ Event-driven snippet (Pub/Sub, Storage trigger) ────────→ Cloud Run functions
│  └─ Managed web app, supported runtime, no Docker ──────────→ App Engine Standard
└─ YES
   ├─ Single stateless container, scale-to-zero, request/event ─→ Cloud Run
   └─ Orchestration / many services / mesh / DaemonSets
      ├─ Hands-off, pay per pod ────────────────────────────────→ GKE Autopilot
      └─ Node-level control / custom node config ───────────────→ GKE Standard

Serverless & GKE

ServiceScale to 0Best for
Cloud Run functions (was Cloud Functions)YesEvent-driven glue, lightweight HTTP, triggers
Cloud RunYesStateless containers, APIs, any language; revisions + % traffic split for canary/rollback
App Engine StandardYesManaged web apps, fast cold start, traffic splitting, spikes
App Engine FlexibleNo (min 1)Custom runtimes, background processes, large deps
GKE Autopilot—Full K8s API, no node mgmt, billed per pod request
GKE Standard—Node-level control, DaemonSets, custom hardware, bin-packing

Scaling on GKE: HPA scales replicas (supports custom/external metrics like RPS via Cloud Monitoring); VPA resizes pods; cluster autoscaler adds nodes when pods are unschedulable. Regional clusters = HA control plane + nodes across zones.

Compute Engine

Pricing & licensing

  • Spot VMs (replaced Preemptible): up to ~60–91% off, reclaimable → fault-tolerant batch in a MIG, checkpoint to GCS.
  • CUD: commit 1/3 yr → steady 24/7. Flexible CUD now spans CE/GKE/Cloud Run.
  • Sustained Use Discounts: automatic, no commitment.
  • Sole-tenant nodes: dedicated physical host → BYOL per-core / isolation. (Confidential VM ≠ isolation — it encrypts memory but shares hardware.)
  • Custom machine types: odd vCPU:RAM ratio.

Managed Instance Groups

  • Instance template = immutable blueprint (required).
  • Autohealing recreates failed VMs (health-check driven).
  • Regional MIG spans zones → survives a zone failure.
  • Rolling updates / canary; autoscaling on CPU, LB capacity, schedule, or Monitoring metric.
  • Stateful MIG preserves per-instance disks/IPs.
  • Compact placement policy → lowest inter-node latency (HPC).

Keyword → service

  • "Lift-and-shift / existing license / OS control"Compute Engine
  • "Run code, scale to zero, single container"Cloud Run
  • "Microservices / mesh / DaemonSets / portable"GKE (Anthos)
  • "Reduce ops, full K8s API, pay per pod"GKE Autopilot
  • "Managed web app, traffic splitting, no Docker"App Engine Std
  • "Fault-tolerant batch, cheapest"Spot VMs in MIG
  • "Steady 24/7 predictable, lowest cost"CUD
  • "Physical isolation / per-core BYOL"Sole-tenant
Traps: App Engine Flexible & Cloud Run w/ min-instances don't scale to zero. Spot VMs never for stateful/always-on. "Recreate failed VMs" = MIG autohealing. "Scale on requests-per-second" = HPA custom/external metric, not cluster autoscaler.
Domain · Storage & Databases

Storage & Databases

The heaviest-tested domain. Database selection is pattern-matching on four axes: relational vs NoSQL, regional vs global, OLTP vs OLAP, and scale.

Database selection matrix

ServiceModelScopePick when (keywords)
Cloud SQLRelationalRegionalManaged MySQL/Postgres/SQL Server; lift-and-shift OLTP
AlloyDBPostgres / HTAPRegionalPostgres but need speed/scale; analytics on transactional data; Oracle→OSS
SpannerRelational, distributedGlobalglobal + strong consistency + relational + horizontal scale; 99.999% multi-region
Firestore (Native)Document NoSQLRegional/MultiMobile/web app, real-time sync, offline SDK
BigtableWide-columnRegionaltime-series / IoT / high throughput / low-latency key lookups
BigQueryColumnar OLAPMulti-regionanalytics / warehouse / BI / ad-hoc SQL over TB–PB
MemorystoreIn-memory (Redis)RegionalCache / session / sub-ms / leaderboard
#1 trap — Spanner over-selection: only when global + strong consistency + relational + horizontal. Single-region relational = Cloud SQL. BigQuery = analytical SQL; Bigtable = operational per-key reads/writes — never swap them.

HA, scaling & data movement

  • Cloud SQL HA = regional synchronous standby + auto failover. Read replicas ≠ HA (scale reads).
  • BigQuery cost control: capacity/slot commitments for predictable spend vs on-demand per-bytes-scanned. Partition + cluster to cut bytes scanned.
  • DMS = migrate to Cloud SQL/AlloyDB (minimal-downtime CDC). Datastream = ongoing CDC into BigQuery. Dataflow = managed Beam ETL. Cloud Composer = Airflow DAG orchestration.

Domain · Networking

Networking

VPC structure, how you connect to on-prem, and which load balancer. The hybrid-connectivity ladder and the LB matrix are high-yield.

VPC fundamentals

  • VPC = global; subnets = regional. Custom mode for production. Secondary ranges = GKE Pods/Services (alias IPs).
  • Firewall rules: lower priority number wins (default 1000); implied deny-ingress + allow-egress; target by service account (preferred) or tag. Hierarchical policies apply org/folder-wide.
  • Shared VPC: central host project owns subnets/firewall, service projects deploy in. VPC Peering: two VPCs as equals, non-transitive. Network Connectivity Center: hub-and-spoke transitive connectivity.

Hybrid connectivity ladder

OptionBandwidthSLAUse when
HA VPN~3 Gbps/tunnel99.99%*Encrypted, low cost, quick (*redundant tunnels)
Partner Interconnect50 Mbps–50 Gbps99.9/99.99%Mid bandwidth, no colo presence
Dedicated Interconnect10–100 Gbps99.9/99.99%High bandwidth, private, has colo
Direct/Carrier Peeringup to 10 GbpsNoneReach only Google public APIs at edge
Trap: Interconnect is not encrypted by default. Need encryption → HA VPN over Interconnect / MACsec. Hybrid private routing with redundancy → Interconnect + Cloud Router (BGP). VPC Peering cannot reach on-prem.

Load balancing & private access

  • "Global HTTP(S), URL routing, CDN, WAF, anycast IP"Global Ext Application LB
  • "Preserve client source IP / UDP / non-TCP"Passthrough Network LB
  • "TLS offload, not HTTP, custom port, global"Proxy Network LB
  • "Balance between internal app tiers"Internal LB
  • "No-external-IP VM → reach Google APIs"Private Google Access
  • "Private VM outbound to internet only"Cloud NAT
  • "Private access to SaaS / published service"Private Service Connect

Global = Application LB & Proxy Network LB only; Passthrough Network LB is regional. Cloud CDN attaches to the External Application LB. Cloud DNS supports public + private zones and routing policies.

Domain · Security & IAM

Security & IAM

Least privilege everywhere, perimeters around data, the right key tier. Know who can do what (IAM) vs what may be configured (Org Policy) vs the data boundary (VPC-SC).

IAM & identity

  • Hierarchy Org → Folder → Project → Resource; policy inherited downward, the union of bindings. A child can't subtract a parent grant → use IAM Deny policy, Org Policy, or VPC-SC.
  • Roles: Basic (avoid in prod) → Predefined (preferred) → Custom. Assign to groups. IAM Conditions for attribute/time-based access.
  • Service accounts: attach to VM/GKE + metadata tokens (no key files). Workload Identity Federation = external/AWS/on-prem auth without keys. GKE Workload Identity = pod-scoped identity. Privileged Access Manager = just-in-time, time-bound elevation.

Keys, perimeters & defense

Encryption tier

  • "Control rotation, Google stores keys"CMEK (KMS)
  • "Google must never store the key"CSEK
  • "Keys fully outside Google"Cloud EKM
  • "FIPS 140-2 Level 3 hardware keys"Cloud HSM
  • "Store/rotate API keys & secrets"Secret Manager
  • "Google support needs my approval to access"Access Approval + Transparency

Access & data protection

  • "Block exfiltration w/ stolen creds"VPC Service Controls
  • "WAF: SQLi/XSS/DDoS, geo, rate limit"Cloud Armor
  • "Access internal app w/o VPN"IAP / BeyondCorp
  • "Central misconfig/threat view"Security Command Center
  • "Mask / tokenize / discover PII"Sensitive Data Protection
  • "Only signed/attested images deploy"Binary Authorization
Trap: Cloud Armor protects L7 web traffic at the edge — to stop data exfiltration even with valid credentials use VPC Service Controls. EU residency → Org Policy gcp.resourceLocations + Assured Workloads. Note: SCC Enterprise tier sunsets May 2027 → Premium.

Disaster recovery

RPO = acceptable data loss · RTO = acceptable downtime.

NeedStrategy

Domain · GenAI & ML (new in v6.1)

Generative AI & Vertex AI

New sections 2.4 (Vertex AI end-to-end ML) and 2.5 (pre-built APIs). Three of the four case studies are GenAI-centric, so this is high-yield. The core skill: pick the right model/API/service for the task.

Foundation models — when to pick each

ModelUse it when the scenario says…
GeminiSummarize/analyze/extract over multimodal content (text+image+video+audio+PDF); reasoning; build an agent / function-calling; code
ImagenGenerate or edit images — product/marketing visuals, inpainting, background removal
VeoGenerate video from text/image prompts
Chirp / Speech-to-TextTranscribe/understand speech at scale (use the dedicated API for pure transcription)
Key distinction: "generate an image/video" → Imagen/Veo. "Understand / summarize / extract / converse over media or docs" → Gemini.

Grounding, RAG & search

NeedService
Grounded search / Q&A over our documents (managed, ACL-aware) — default RAG answerVertex AI Search
Managed RAG but need a customizable pipelineVertex AI RAG Engine
Semantic / embedding similarity over millions of vectorsVertex AI Vector Search
Product search / recommendations for an online storeVertex AI Search for commerce (Discovery AI)

RAG pattern: ingest & chunk → embed → index in Vector Search → retrieve top-k → augment + ground → Gemini answers.

Conversational, safety & pre-built APIs

  • "Chatbot / replace the IVR / self-service agent"Conversational Agents (Dialogflow CX)
  • "Help live human agents on calls"Agent Assist
  • "Modernize the whole contact center"Contact Center AI
  • "Screen GenAI prompts/responses for safety"Model Armor
  • "Explain model decisions / regulatory"Explainable AI
  • "Detect model drift in production"Model Monitoring
  • "Extract fields from invoices / PDFs / forms"Document AI
  • "Sentiment / entities from text"Natural Language API
  • "ML on data already in BigQuery, via SQL"BigQuery ML
  • "End-to-end MLOps, online endpoints"Vertex AI (Pipelines/Registry/Endpoints)
Rule: a purpose-built API beats Gemini for standard structured tasks (transcribe, translate, OCR/extract, sentiment) — cheaper, faster, structured output. Gemini beats an API for cross-modal understanding, summarization/generation, reasoning, agents.
Cross-cutting · WAF & Migration

Well-Architected Framework & Migration

Both now examined. The WAF is the lens for "best balance" questions; migration covers the strategies, phases, and tooling.

The six WAF pillars

PillarQuestion signal → answer pattern
Operational Excellence"automate deploys / reduce toil" → IaC, CI/CD, managed services
Security, Privacy & Compliance"least privilege / protect data / HIPAA-GDPR / no exfiltration" → IAM, CMEK, DLP, VPC-SC
Reliability"must stay up / meet RTO-RPO / survive failure" → multi-zone/region, SLOs, tested DR
Cost Optimization"reduce/control spend" → autoscaling, rightsizing, CUDs/spot, budgets
Performance Optimization"slow / spikes / scale" → autoscaling, CDN/caching, right machine, load testing on real data
Sustainability"greener / lower carbon" → low-carbon regions, serverless, higher utilization

Migration: the Rs, phases & tools

StrategyTool / target
Rehost (lift & shift)Compute Engine via Migrate to Virtual Machines
Replatform (move & improve)DB → Cloud SQL/AlloyDB via DMS
Refactor (re-architect)Containerize → GKE/Cloud Run via Migrate to Containers; strangler-fig pattern
Repurchase / RetireSaaS replacement / decommission
Hybrid / multicloud one control planeGKE Enterprise (Anthos) + Config Management

Phases: Assess → Plan → Migrate → Optimize. Migration Center (now required) does discovery, dependency mapping, TCO. Prefer phased waves over big-bang. Huge data + slow link → Transfer Appliance (offline); good link → Storage Transfer Service (online).

Likely exam questions — tap to reveal

Minimum 99.9% availability for apps already running in GKE — best approach?
Regional GKE cluster behind a Global HTTP(S) Load Balancer. Regional = multi-zone redundancy; a zonal cluster can't meet the SLA.
Secure, high-bandwidth, low-latency link to keep legacy insurance integrations reachable?
Dedicated Interconnect. Highest-performance private connectivity; HA VPN can't match sustained throughput/latency.
Alerts are emailed and ignored; they want centralized proactive visibility — implement what?
Cloud Monitoring dashboards + alerting policies with Cloud Logging. Native, centralized, proactive.
Existing users are in Microsoft Active Directory — manage GCP identities with least effort?
Google Cloud Directory Sync into Cloud Identity, federate SSO. Reuse the existing source of truth instead of recreating users.
Manage container workloads consistently across on-prem and Google Cloud?
GKE Enterprise (Anthos) fleet with Config Management. One consistent config/policy plane across hybrid.
Migrate many on-prem VMware VMs with minimal downtime?
Migrate to Virtual Machines (M2VM). Automated replication + low-downtime cutover; DMS is for databases only.
